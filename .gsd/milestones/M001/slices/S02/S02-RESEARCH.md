# S02: Wallet Connect and Anchor Client — Research

**Date:** 2026-03-19

## Summary

S02 wires Solana wallet connectivity into the Next.js 15 App Router and sets up the Anchor program client for both `gherkin_pay` and `gherkin_pay_hook`. The wallet adapter ecosystem is mature and well-documented. The official Solana guide confirms the pattern: a `"use client"` provider wrapping `ConnectionProvider` → `WalletProvider` → `WalletModalProvider`, with `wallets={[]}` letting Wallet Standard auto-detect Phantom/Solflare. The Anchor client needs IDL JSON files — these don't exist yet and must be generated via `anchor build`.

The main risks are: (1) `anchor build` must succeed to produce IDL/types — this is the prerequisite for everything; (2) Next.js webpack config may need `externals` for Node.js modules pulled in transitively by Solana packages (`pino-pretty`, `lokijs`, `encoding`); (3) the wallet adapter's default CSS (`@solana/wallet-adapter-react-ui/styles.css`) may clash with GherkinPay's design system — we should import it but may need overrides.

## Recommendation

Build in this order: (1) Generate IDL via `anchor build`, copy IDL JSON and TypeScript types into `app/web`; (2) install Solana/wallet/Anchor deps in `app/web`; (3) create `WalletContextProvider` client component; (4) create constants and Anchor program hook; (5) wire provider into root layout and replace the placeholder Button with `WalletMultiButton`; (6) update `next.config.mjs` with webpack externals if build fails; (7) verify with `bun run build`.

### Build Order

1. **Generate IDL + types** — `anchor build` in project root. This produces `target/idl/gherkin_pay.json`, `target/types/gherkin_pay.ts`, and equivalents for the hook program. Copy these into `app/web/src/idl/` and `app/web/src/types/`. This unblocks everything else — without IDL, the Anchor client can't instantiate.

2. **Install deps** — In `app/web`: `@solana/web3.js`, `@solana/wallet-adapter-base`, `@solana/wallet-adapter-react`, `@solana/wallet-adapter-react-ui`, `@coral-xyz/anchor`. Pin to `@solana/web3.js@^1.98.0` (v1.x, not v2) since both wallet adapter and Anchor depend on v1.

3. **Constants + env** — Create `lib/constants.ts` with program IDs and RPC endpoint (from env var). Add `NEXT_PUBLIC_SOLANA_RPC_URL` to `env.js` schema. Default to `https://api.devnet.solana.com`.

4. **Wallet provider** — Create `components/wallet-provider.tsx` as a `"use client"` component. Use `wallets={[]}` (Wallet Standard auto-detects). Import `@solana/wallet-adapter-react-ui/styles.css`. Wrap in root `layout.tsx` inside `TRPCReactProvider`.

5. **Anchor hook** — Create `lib/anchor.ts` with `useAnchorProgram()` that uses `useConnection()` and `useAnchorWallet()` from wallet adapter, creates `AnchorProvider`, returns typed `Program` instances. Must handle disconnected wallet gracefully (return null programs).

6. **Wallet button in header** — Replace placeholder `<Button>Connect Wallet</Button>` in `(console)/layout.tsx` with `WalletMultiButton` from `@solana/wallet-adapter-react-ui`. This component is a `"use client"` import but the console layout currently isn't a client component — may need to extract the wallet button to a small client component wrapper.

7. **Build verification** — `bun run build`. If webpack errors about `pino-pretty`/`lokijs`/`encoding`, add `config.externals.push(...)` to `next.config.mjs`.

### Verification Approach

- `anchor build` exits 0, `target/idl/gherkin_pay.json` and `target/types/gherkin_pay.ts` exist
- `bun run build` in `app/web` exits 0 (catches polyfill/import/type errors)
- `bun run typecheck` passes (must run after build per KNOWLEDGE.md)
- IDL JSON files parseable and contain expected program addresses
- `WalletMultiButton` renders in the console sidebar (visible in build output as compiled route)
- No `@/` imports in any new file (all use `~/`)

## Don't Hand-Roll

| Problem | Existing Solution | Why Use It |
|---------|------------------|------------|
| Wallet connection UI (modal, multi-button) | `@solana/wallet-adapter-react-ui` `WalletMultiButton` | Standard Solana UX, auto-detects wallets via Wallet Standard, handles connect/disconnect/switch |
| Wallet auto-detection (Phantom, Solflare) | Wallet Standard (`wallets={[]}`) | No need to import individual wallet adapters — standard-compliant wallets are auto-detected |
| Anchor provider + program client | `@coral-xyz/anchor` `AnchorProvider` + `Program` | Type-safe account fetching, PDA derivation, instruction building from IDL |
| IDL generation | `anchor build` | Generates JSON IDL + TypeScript types from Rust program source |

## Constraints

- `"use client"` boundary required for all wallet hooks (`useWallet`, `useConnection`, `useAnchorWallet`) — server components cannot use them
- `(console)/layout.tsx` is currently a server component — the wallet button must be extracted to a client component
- `app/web/tsconfig.json` uses `~/` path alias — all new files must use `~/` imports, not `@/`
- `@solana/web3.js` must be v1.x (not v2) — wallet adapter and Anchor both peer-depend on v1
- Anchor CLI 0.31.1 is installed; IDL format is the 0.30+ "spec 0.1.0" format with `address` field and `discriminator` arrays
- `bun run typecheck` must run after `bun run build` (KNOWLEDGE.md rule — Next.js generates `.next/types/**/*.ts` during build)

## Common Pitfalls

- **Next.js webpack externals** — Solana packages transitively import `pino-pretty`, `lokijs`, `encoding` which are Node.js-only. Must add `config.externals.push("pino-pretty", "lokijs", "encoding")` to `next.config.mjs` or build will fail with module resolution errors.
- **Wallet adapter CSS import location** — `@solana/wallet-adapter-react-ui/styles.css` must be imported in the wallet provider client component (not in a server component or globals.css `@import`). If imported wrong, the modal renders unstyled.
- **`anchor build` requires Rust/Solana toolchain** — If BPF tools aren't installed, the build will fail. Fall back to manually creating IDL JSON from the Rust source if needed.
- **Console layout is server component** — Importing `WalletMultiButton` directly will fail. Must create a thin `"use client"` wrapper component (e.g., `wallet-button.tsx`) and import that instead.
- **IDL `resolveJsonModule` + verbatimModuleSyntax** — The tsconfig has `verbatimModuleSyntax: true`. Importing JSON with `import idl from "./idl.json"` needs a type assertion. Use `import idl from "~/idl/gherkin_pay.json"` and cast: `idl as GherkinPay` where `GherkinPay` is the generated type.

## Open Risks

- `anchor build` may fail if BPF/SBF toolchain isn't fully set up on this machine. If so, the IDL files must be hand-created from the Rust source — doable but tedious. The test file provides account shapes as a cross-reference.
- Turbopack (`next dev --turbo`) may handle webpack externals differently than the production build. Dev might work while `bun run build` fails, or vice versa. Test with `bun run build` as the source of truth.
